00 — Orientation

The whole thing, in sixty seconds

Strip away the code and this is one sentence: each month, in each country, buy the stocks that have been going up — but only while the market itself is healthy — size them so the risk is steady, and report the combined result in one currency.

That is a cross-sectional momentum strategy, the single most-replicated anomaly in finance (Jegadeesh & Titman, 1993). The interesting part isn't the idea — it's everything built around the idea to stop it lying to you: no-lookahead accounting, costs that are never switched off, one weight function shared by the simulation and the live book, and a test suite that fails the build if any of those slip.

What you'll learn — engineering

Data-driven design, a single source of truth, idempotency, fail-safe persistence, invariants enforced as executable tests, and how to structure a system so adding a market is one line of config.

What you'll learn — trading

Momentum & trend filters, inverse-vol and volatility targeting, transaction costs & tax asymmetry, currency P&L, survivorship bias, and how professionals measure whether an "edge" is real or a mirage.

Lesson 00 · the meta-lesson

In quant trading, the strategy is the easy 20%. The 80% that separates a toy from a system is the plumbing that keeps you honest: accounting for time correctly, never hiding a cost, and building so that the thing you test is exactly the thing you run. This guide is mostly about that 80%.

01 — Architecture

One Region record to rule them all

The system runs three live markets — London (FTSE), the US, and Australia (ASX) — plus a fourth (Canada/TSX) scaffolded but deliberately unfunded. The naïve way to do this is to copy the ASX code three times and edit the ticker suffixes. That path guarantees three subtly different bugs.

Instead, everything that differs between markets is data, not code. A single frozen Region dataclass captures the entire difference between London and Sydney: its universe, its regime index, its currency, its fee schedule, its trading calendar, its Yahoo ticker convention, its broker routing, and its price scaling.

regions.py — the difference between two markets is a record
@dataclass(frozen=True)
class Region:
    key: str                 # "ASX"
    currency: str            # local trading currency: AUD / USD / GBP
    index_ticker: str        # regime filter index, e.g. ^GSPC
    yahoo_suffix: str        # ".AX", "", ".L"
    commission_bps: float; min_commission: float; slippage_bps: float
    stamp_duty_bps: float    # tax on BUYS only (UK SDRT); 0 elsewhere
    price_scale: float       # 0.01 turns LSE pence into pounds
    universe: list[str] = field(default_factory=list)
    params: StrategyParams = DEFAULT_PARAMS   # per-region overrides

Adding Canada was, quite literally, one entry in a dictionary. The comment in the code is the whole thesis of the architecture:

“Everything downstream is parameterised by a Region, so adding a fourth market (e.g. TSX or HKEX) is just one more entry here.”

The knobs live in one frozen dataclass too

Every strategy parameter — the momentum window, the number of names to hold, the volatility target — lives in a single immutable StrategyParams. Nothing reads a module-level global; the parameters are always passed in. That is why the exact same signal code runs identically for four markets and in both the backtest and live trading. Immutability (frozen=True) means a parameter set can be shared across threads and sleeves with zero risk of one mutating another's config.

Lesson 01 · engineering

Push variation into data. When the things that differ between cases become records rather than branches, the code that acts on them stays small, uniform, and testable — and the surface area for copy-paste bugs collapses to near zero. "Add a market" should be a config change, not a code change.

02 — The signal

12-1 momentum: buy what's winning, minus last month

The core signal is one line. For each stock, take its total return over roughly the last twelve months — but skip the most recent month:

signals.py — the entire alpha idea in one expression
def momentum_score(prices, p):
    # return over lookback (252d) excluding the most recent skip (21d)
    return prices.shift(p.skip_days) / prices.shift(p.lookback_days) - 1.0

Notice the tool doing the work: .shift(). Every signal is built purely from past prices. This isn't a stylistic choice — it is the load-bearing wall of the whole system, and we'll return to it in Part 05.

Three filters turn a score into a decision

A high momentum score is necessary but not sufficient. A name only enters the book if it clears three gates — and the whole book goes to cash if the market itself is unhealthy:

  • Absolute momentum — the 12-1 return must be positive. Don't buy the "best of a falling universe."
  • Per-stock trend — price must be above its own 200-day moving average. An individual name in a downtrend is disqualified however good its ranking.
  • Regime filter — if the index is below its 200-day MA, the market is risk-off and the sleeve holds 100% cash. This is the crash protection: momentum's worst enemy is a sharp market reversal, and this gate sidesteps most of it.

An optional value factor (long-term reversal — cheap = past losers) can be blended in by cross-sectional percentile rank. It's off by default, but its design is instructive: it is negatively correlated with momentum by construction, which is exactly what makes it a diversifying signal rather than a redundant one.

Lesson 02 · trading

An edge is a filter, not a prediction. The system never claims to know a stock will rise. It ranks, gates on trend, and defers to the regime. The filters do more for the risk profile than the ranking does for the return — crash protection (the index MA gate) is worth more than a cleverer score.

03 — Position sizing

Steady risk, not steady money

Picking the names is half the job. How much of each to hold is the other half, and it's where the risk discipline lives. Sizing happens in two stages.

Stage 1 — Inverse-volatility weights

Calmer stocks get more capital; wilder stocks get less. Each weight is proportional to 1/volatility, then capped at 15% per name so nothing dominates. If the cap pushes the book above 100% invested, it's renormalised back down — but never levered up from a cap.

Stage 2 — Volatility targeting

The whole book is then scaled toward a 12% annualised volatility target. In calm markets the book scales up; in turbulent markets it scales down. The point is that risk stays roughly constant through time, rather than the dollar amount invested.

Portfolio volatility is estimated with a constant-average-correlation approximation — you don't need a full covariance matrix, just an assumed average pairwise correlation ρ:

strategy.py — vol targeting, capped and never levered past gross
# var ≈ (1−ρ)·Σ(wᵢσᵢ)²  +  ρ·(Σ wᵢσᵢ)²
port_var = (1.0 - rho) * (wv**2).sum() + rho * (wv.sum() ** 2)
port_vol = float(np.sqrt(max(port_var, 0.0)))

scale = min(p.target_vol / max(port_vol, 1e-9), p.max_vol_scale)
w = weights * scale
if w.sum() > p.max_gross:          # gross exposure never exceeds 100%
    w = w * (p.max_gross / w.sum())
Lesson 03 · trading

Volatility targeting is the closest thing to a free lunch in systematic trading. It doesn't improve your average return, but it stabilises the ride — cutting exposure into turbulence and adding it into calm. Steady risk compounds better than steady capital, and it makes the strategy psychologically survivable.

04 — Single source of truth

One weight function, called by everything

Here is the most important architectural decision in the entire system, and it is almost invisible. Selection and sizing live in exactly one function — strategy.compute_targets() — and both the backtester and the live paper-trading engine call it.

strategy.py — the only place target weights are ever born
def compute_targets(prices, index_prices, p, asof=None, eligible=None):
    scores  = sig.momentum_score(prices, p).loc[asof]
    trend   = sig.stock_trend_ok(prices, p).loc[asof]
    vols    = sig.realised_vol(prices, p).loc[asof]
    risk_on = sig.index_risk_on(index_prices, p)...loc[asof]
    raw = sig.select_portfolio(scores, trend, vols, risk_on, p, ...)
    return vol_target(raw, vols, p)     # selection + sizing, one path

Why does this matter so much? Because the number-one way backtests betray you is drift between simulation and reality. You test one version of the sizing logic, then live trading grows a slightly different copy — an extra cap here, a rounding tweak there — and now your backtest is describing a strategy you don't actually run. The results become fiction.

This system makes that class of bug impossible by construction: there is no second copy to drift. And it doesn't just trust that — it proves it in the test suite, at two levels:

tests/test_consistency.py — the guarantee is executable
# Layer 1 — source-level: neither engine may bypass the shared path
assert "select_portfolio" not in inspect.getsource(backtest)
assert "select_portfolio" not in inspect.getsource(paper_trade)

# Layer 2 — numeric golden equality: the weights each engine ACTS on
# equal compute_targets' output bit-for-bit. A sneaky second sizing
# path (an ADV cap trimmed post-hoc) can't slip past a text grep.
Lesson 04 · engineering

Don't Repeat Yourself is a correctness property, not a tidiness preference. When two things must agree, don't write them twice and hope — write them once and route both callers through it. Then write a test that fails the build if anyone forks it. An invariant a human has to remember is an invariant that will eventually break.

05 — The backtest

No lookahead: decide at t, trade at t+1

A backtest's only job is to answer "what would have happened?" honestly. The single easiest way to fake a brilliant one is to peek — to let a decision on Monday quietly use Monday's closing price that you wouldn't have known until the day was over. The system's first invariant exists to forbid this:

A decision at time t uses only data available at t, and is acted on at t+1.

The backtester separates two clocks. Decisions happen on month-end rebalance dates, using data up to and including that day. Execution and mark-to-market happen every trading day — and crucially, yesterday's decided target becomes today's trade:

backtest.py — the t+1 execution loop (abridged)
for i in range(1, len(dates)):
    today, yday = dates[i], dates[i-1]
    if pending is not None:              # apply YESTERDAY's target at TODAY's prices
        delta = pending - current_w
        cost  = turnover*cost_rate + buy_turnover*stamp_rate
        current_w = pending
    r = (current_w * day_rets).sum() - cost
    equity.append(equity[-1] * (1 + r))
    if yday in weight_schedule:        # schedule computed at t, fires at t+1
        pending = weight_schedule[yday]

The drawdown circuit breaker

Layered on top is a portfolio-level safety valve: if equity falls more than 25% from its peak, the book liquidates to cash and sits out a cooldown (~one month) before re-entering. It's a coarse instrument — but momentum strategies die in fast crashes, and a hard stop is a blunt, reliable defence. Notice it obeys the same discipline: the breach is detected at the close and the flatten executes the next day.

Lesson 05 · engineering + trading

Model time as carefully as you model money. Off-by-one-day errors don't crash — they silently inflate returns and make a losing strategy look like a winner. The fix is structural: separate the moment of decision from the moment of action, and let data flow only forward. Then pin it with a test that recomputes a signal on truncated history and demands the same answer.

06 — Frictions

Costs are always on — including the tax nobody models

A backtest with costs switched off is a fantasy generator. Momentum is a relatively high-turnover strategy — it rebalances monthly and chases moving names — so it is especially sensitive to frictions. This system charges three, every single rebalance:

Commission

max(floor, notional × bps). The floor matters enormously for small accounts — a $1 minimum on a $50 trade is a 2% cost. This is why the project runs a deliberate micro account: to show fee drag, not hide it.

Slippage

A per-side price impact in basis points, wider for thinner markets (ASX 10bps) than deep ones (US 5bps). Modelled as an adverse fill, not a fee.

UK stamp duty: the asymmetry that changes the answer

The UK charges 0.5% Stamp Duty Reserve Tax on share purchases only — not sales, and not in any other market. Most toy backtests ignore this entirely. For a high-turnover UK momentum book it's a material, recurring, one-sided drag, so the system models it explicitly and separately from symmetric costs:

fees.py — asymmetric by design
def stamp_duty(region, buy_notional):
    # tax charged on the BUY notional only (0 for sells / non-UK)
    return max(buy_notional, 0.0) * region.stamp_duty_bps / 1e4

Lesson 06 · trading

The gap between a backtest and reality is mostly costs. Turnover, floors, slippage, and jurisdiction-specific taxes are not footnotes — they are the difference between a strategy that survives contact with a broker and one that quietly bleeds. Model the frictions that are asymmetric or fixed especially carefully; those are the ones that ambush small accounts.

07 — Portfolio & currency

Three books, three currencies, one AUD number

Each sleeve trades and compounds entirely in its own local currency — an invariant: you never mix GBP and USD inside a sleeve. Currency only enters at the very top, when the portfolio layer combines everything into the base reporting currency (AUD). And when it does, it captures the currency move as real P&L:

portfolio_backtest.py — a foreign sleeve's return in AUD includes the FX move
# m = base units per 1 local unit (AUD per GBP, etc.)
fx_ret = m / m.shift(1) - 1.0
return (1 + local_rets) * (1 + fx_ret) - 1.0     # stock P&L compounded with FX P&L

A British stock can rise 3% while the pound falls 2% against the Aussie dollar — an Australian investor's real return is neither number, it's the product. Ignoring this is a classic error that makes multi-market backtests wrong in ways that are invisible until you actually wire real money across a border.

Allocation, diversification, and an honest benchmark

  • Equal thirds, rebalanced. Capital splits evenly across sleeves and is trued back to target on a cadence — paying a small FX spread on the cash that crosses currencies (again: the cost is never free).
  • Diversification is measurable. Because the three markets aren't perfectly correlated, the combined book's volatility is lower than any single sleeve's — the synthetic sanity check confirms it, which is how you know the plumbing is right.
  • The benchmark is fair. Performance is judged against an equal-weight buy-and-hold of the three regional indices, converted to AUD the same way — plus beta, Jensen's alpha, tracking error and information ratio. You only get to claim skill relative to the cheap alternative.
Lesson 07 · engineering + trading

Currency is a position, not a display setting. Keep each unit of the system in one honest currency and convert only at a single, well-defined boundary — capturing the conversion's P&L, not papering over it. The same discipline generalises: don't mix units (currencies, timezones, price scales) inside a component; convert at the edges, explicitly.

08 — Backtest → live

The jump from simulation to a real, persistent book

A backtest deals in fractional weights and continuous equity. A live book cannot — it buys whole shares, pays a commission floor on every ticket, holds state across days on a machine that might crash mid-run, and must never trade twice on the same signal. The paper-trading engine is where the strategy meets these hard edges. It's the largest, most defensive module in the system, and it teaches the most about production engineering.

Whole shares & the tyranny of the floor

Targets are fractional; reality is integer. Converting is a floor division — you can only buy shares you can fully afford — and tiny "dust" adjustments are skipped entirely, because paying a $5 commission to rebalance a $30 drift is lighting money on fire:

paper_trade.py — fractional target → integer lot
desired[t] = int((equity * w) / price)          # whole shares only
...
if notional < dust and tgt != 0:                # skip dust; always allow a full exit
    continue
fee  = fees.commission(region, notional)
duty = fees.stamp_duty(region, notional) if delta > 0 else 0.0

Very small accounts can't even hold the full book in whole shares, so there's an explicit micro mode that concentrates into a few affordable names rather than pretending to own fractional slices — an honest degradation instead of a silent lie.

P&L comes from the fills ledger — the one source of truth

This is a subtle and excellent decision. The book stores positions and cash, but realised P&L is never stored as a running total. Instead it's derived from the immutable log of actual fills, replayed through a FIFO lot-matching engine — exactly like a broker's trade confirmations. Each sale is stamped with the real cost basis of the specific lots it consumed:

paper_trade.py — realised P&L reconstructed from the trade record, not a cached counter
lots, _ = pnl.build_lots([t for t in trade_log if t["region"] == region.key])
...
if delta > 0:
    pnl.add_lot(lots, key, delta, fill, fee+duty, today)   # a buy opens a lot
else:
    r = pnl.consume(lots, key, abs(delta), fill, fee)     # a sell eats FIFO lots
    trade["realized"] = round(r["net"], 2)             # actual gain/loss on this sale

The code comment says it best: cost-basis counters "are NOT the source of truth and nothing reads them… so it can never disagree with the actual trade record." A derived number can't drift from the events that produced it.

Idempotency: safe to run five times a day

A scheduler fires after each regional market close — so the engine might run three times in one day. It must be idempotent: a sleeve rebalances only on the first run of a new calendar month, and merely marks-to-market otherwise. There's even a subtle guard against churning a freshly-funded book: a minimum-gap rule stops an account funded on the 28th from being torn up on the 1st.

Fail-safe persistence

State is the crown jewels — lose it or corrupt it and you've "reset" a real book. The design is belt-and-braces:

  • Two stores. SQLite is the durable, atomic source of truth (WAL mode + a busy-timeout so a manual run and the scheduler serialise instead of clobbering); a per-account JSON file is dual-written as a human-readable fallback that dashboards and CI can glob.
  • Atomic writes. The JSON is written to a temp file, fsynced, then atomically renamed over the target — a crash leaves either the old file or the new one, never a half-written book.
  • Additive migration. Older state files are upgraded on load, never destructively — new fields are added, nothing existing is dropped.
  • Fail safe, don't fail silent. With the validation gate on, a corrupt-but-parseable state file makes the run raise rather than trade on garbage or reset equity. The default is a "shadow mode" that only warns during rollout — a staged, reversible way to turn on a strict check.
storage.py — the canonical torn-write-proof file update (memorise this)
tmp = f"{path}.tmp"
with open(tmp, "w") as f:
    json.dump(state, f, indent=2)
    f.flush(); os.fsync(f.fileno())      # force bytes to disk first
os.replace(tmp, path)                     # atomic on POSIX — no torn-write window
Lesson 08 · engineering

Derive, don't cache; and make the unsafe path the one that fails loudly. Store the immutable facts (the fills) and compute everything else from them — a derived total can never disagree with its inputs. For anything that persists real state, assume the process will be killed mid-write, run twice by accident, and handed yesterday's file format. Idempotency, atomic writes, additive migrations and fail-safe validation aren't paranoia — they're the difference between a demo and a system you can leave running.

09 — Honest measurement

Survivorship bias & the bad-print problem

Two quieter systems exist purely to stop the data from flattering you.

Survivorship bias — the backtest is an upper bound

Today's index members are the survivors. A stock that collapsed and dropped out of the ASX 200 in 2015 isn't in today's universe — so a naïve backtest never "buys" it and never takes its loss. The results are silently inflated. The constituents.py module fixes this with point-in-time membership: at each historical rebalance, only names that were actually in the index that month are eligible, and since-delisted names are included if their prices can be fetched.

Tellingly, the system labels its own output. Without a point-in-time file it falls back to today's universe and says so — it never presents a survivorship-biased number as if it were clean.

Data quality — one bad print corrupts everything downstream

Before compute_targets ever runs, a shared validator screens every price series for staleness, gaps, outliers and impossible moves. Flagged names are dropped from the candidate set. In live trading a flagged name that's already held is frozen — not traded on an untrusted price — rather than dumped at a bad quote. The gate runs in both the backtest and live, so they stay consistent.

Lesson 09 · trading + engineering

Your backtest is a best case, and your data is guilty until proven innocent. Survivorship bias, look-ahead, and bad prints all push results in the flattering direction — the direction you want to believe. The professional instinct is the opposite of the amateur's: assume the impressive number is wrong, and build the machinery to prove it isn't. When you can't, label it honestly.

10 — The FX subsystem

A parallel ecosystem of opinions

Living alongside the equity sleeves is a separate, self-contained foreign-exchange system. It reuses the same principles — no lookahead, costs always on, one shared weight function — but redesigns them for a different world: currency pairs instead of single names, long/short leverage instead of long-only, a fast per-bar cycle instead of a monthly rebalance, and — most interestingly — five competing strategy agents instead of one momentum score.

The ensemble: lean on whoever's currently right

The clever part is how the five opinions are combined. In adaptive mode, each agent is weighted by its own recent risk-adjusted performance on that specific pair — the rolling information ratio of "signal yesterday × return today." Agents that have been right get more say; agents that have been wrong get backed off — but a floor ensures none is ever fully muted, because today's laggard is tomorrow's star when the regime flips. The default blending uses the Hedge algorithm (multiplicative weights), which carries a provable regret bound — a rare thing: a combining rule with a mathematical guarantee it won't do much worse than the best agent in hindsight.

The deep-learning layer — that augments, never replaces

A pure-NumPy neural network sits as an optional sixth agent. Two design choices make it worth studying:

  • A Sharpe-ratio loss, not MSE. The network outputs a position and is trained to directly maximise risk-adjusted return (Deep Momentum Networks, 2019). Predicting the next price accurately is the wrong objective — you don't get paid for low forecast error, you get paid for good positions. Optimise the thing you actually care about.
  • Meta-labeling. A secondary classifier (López de Prado's technique) decides how big a bet to place on the primary signal's side, using triple-barrier labels. Separating "which way" from "how much" is a powerful decomposition.

The honesty is baked in even here: the README states plainly that daily FX is near-random-walk and that this layer is "built to measure honestly, not to overclaim." A small regularised MLP is chosen over an LSTM or Transformer precisely because the evidence says simpler models win on noisy financial data.

The risk layer, and a truth naive caps miss

Turning the ensemble's convictions into positions runs through four sequential controls: vol targeting (with signed weights, so offsetting longs and shorts correctly reduce estimated risk), a per-pair cap, per-asset-class gross caps, and a final gross-leverage cap. The third one encodes a lesson most retail systems get wrong:

"Diversification" inside a correlated cluster is fake diversification. Three "different" cryptos are, to a risk model, essentially one bet; four treasury ETFs are one duration bet; five US mega-caps are one cluster. So the system caps the combined gross exposure of each class, not just each position. It's a risk policy, not a market view — and it's the kind of thing that only shows up when you take correlation seriously.

And a neat latency trick

Because the live cycle should stay fast no matter how many years of history pile up, compute_targets(fast=True) trims the price panel to exactly the number of recent bars that can still mathematically affect the latest weight (EMA settle time plus the rolling-window chain). A test proves the trimmed answer is bit-for-bit identical to recomputing on the full archive — so you get bounded, flat latency with zero loss of correctness.

Lesson 10 · engineering + ML

An ensemble of simple, diverse opinions beats one clever model — and you should optimise the objective you're actually paid for. Five interpretable agents you can reason about, weighted by live performance with a floor, is more robust and more debuggable than a monolith. And when you do reach for ML, train it on your real goal (risk-adjusted return), not a convenient proxy (price error).

11 — Not fooling yourself

The hardest problem in quant is self-deception

If you try 500 strategy variants, one will look brilliant by pure luck. The FX subsystem takes this threat seriously enough to build a whole validation apparatus whose job is to disprove edges. This is the most intellectually honest part of the codebase, and the most transferable.

Deflated Sharpe Ratio

Adjusts a strategy's Sharpe downward for how many strategies you tried and how non-normal the returns are. A Sharpe of 2 found after 200 attempts is not a Sharpe of 2. It makes "we found something" a falsifiable claim.

Probability of Backtest Overfitting

Via combinatorially-symmetric cross-validation (CSCV): how often does the "best" in-sample configuration underperform out-of-sample? A high PBO means your selection process is fitting noise.

Purged & embargoed walk-forward

Ordinary cross-validation leaks the future into the past when your labels span multiple bars. The fix is to purge training samples whose outcome windows overlap the test set, and embargo a gap after each test fold. Scalers are fit on the training fold only. The result: every out-of-sample prediction is genuinely out-of-sample — no subtle leakage inflating the score.

Two more honesty guardrails run throughout:

  • The research agent mostly says "no." research.py systematically searches candidate edges (mean-reversion, breakouts, stat-arb pairs) and judges each with DSR and PBO — "it mostly disproves edges; that's the point."
  • Synthetic ≠ performance. Every synthetic run is labelled a pipeline test only. The generator has unrealistically persistent trends; the CLI itself warns you never to read its numbers as returns. And a whole doc (HFT_REALITY.md) exists to argue honestly why true high-frequency trading is out of scope rather than faking it.
Lesson 11 · the deepest lesson

The market's job is to find the flaw in your reasoning; your job is to find it first. Multiple-testing inflation, leakage, and overfitting are not exotic risks — they are the default outcome of searching for patterns in noisy data. Real rigor means building tools that penalise you for looking, that quarantine the future from the past, and that are happiest when they kill your favourite idea.

12 — Craft & process

Invariants that can't be quietly broken

A system this careful is only as good as its ability to stay careful as it changes. The answer isn't discipline (humans forget) — it's turning every rule into an executable check.

The invariants are written down — and enforced

The project's CLAUDE.md lists six invariants: no lookahead, costs always on, one weight function, whole shares, "synthetic is never performance," and "each sleeve trades in its local currency." Crucially, these aren't aspirations in a doc — each is pinned by tests:

  • No lookahead → tests recompute a signal on truncated history and assert it's unchanged.
  • One weight function → the consistency tests grep both engines and check numeric bit-for-bit equality.
  • Behavioural regression gate → CI runs the synthetic backtest and compares it to a committed baseline, catching accidental lookahead/cost/sizing drift a unit test wouldn't. It's updated only deliberately.

Shipping with the same discipline as trading

Even the roadmap is run like a trading desk. The product/ backlog is an evidence-first state machine: every feature is researched, critiqued through four independent lenses (Product Owner, Data Scientist, Chief Engineer, Algo Trader), and can't reach "ready" until it has acceptance criteria and a staged rollout plan. The backlog is machine-checked in CI. As the process doc puts it — it's "the same discipline the trading side applies to money — hypothesis, evidence, staged exposure, guardrails, rollback — applied to shipping."

Lesson 12 · engineering

An invariant a human must remember is already broken. Write the rule as a test that fails the build. The best codebases don't rely on care — they make carelessness mechanically impossible to merge. And the same rigor you apply to the risky thing (money) is worth applying to the other risky thing (change).
